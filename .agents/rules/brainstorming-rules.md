# 아이디어를 설계로 구체화하는 브레인스토밍

## 개요

자연스러운 협업 대화를 통해 아이디어를 온전한 형태의 설계와 사양(Spec)으로 발전시키도록 돕습니다.

먼저 현재 프로젝트의 컨텍스트를 파악하고 질문을 하나씩 던져 아이디어를 구체화합니다.
무엇을 만들지 완전히 이해했다면 설계를 제시하고 사용자의 승인을 받습니다.

<HARD-GATE>
설계를 제시하고 사용자가 이를 승인하기 전까지는 어떠한 구현 스킬도 호출하지 말고,
코드를 작성하거나 어떠한 구현 작업도 하지 마십시오.
아무리 간단해 보이는 작업이라도 예외 없이 적용됩니다.
</HARD-GATE>

## 안티 패턴: "너무 간단해서 설계가 필요 없다"

모든 작업은 이 과정을 거쳐야 합니다. 단순해 보이는 작업일수록 검토되지 않은 가정으로
인해 가장 많은 리소스가 낭비됩니다. 단순한 경우 설계안은 몇 문장으로 짧을 수 있지만,
반드시 제시하고 승인을 받아야 합니다.

## 체크리스트

다음 항목 각각에 대해 작업을 생성하고 순서대로 완료해야 합니다:

1. **프로젝트 컨텍스트 파악** — 아래 "컨텍스트 파악 순서"에 따라 진행합니다.
2. **명확한 질문 던지기** — 한 번에 하나씩 질문하여 목적, 제약 조건, 성공 기준을 파악합니다.
3. **2~3가지 접근 방식 제안** — 각 방식의 장단점과 권장 사항을 포함합니다.
4. **설계안 제시** — 복잡도에 따라 섹션을 나누어 제시하며, 각 섹션마다 사용자의 승인을 받습니다.
5. **설계 문서 작성** — `docs/plans/YYYY-MM-DD-<주제>-design.md`에 저장하고 커밋합니다.
6. **구현으로 전환** — `writing-plans` 스킬을 호출하여 세부 구현 계획을 생성합니다.

## 컨텍스트 파악 순서

브레인스토밍 시작 시 반드시 아래 순서로 파악합니다:

### 1단계: 고정 정보 파악 (`docs/memory/project-context.md`)

- 기술 스택, 아키텍처 원칙, 코딩 컨벤션 확인
- 소스 코드 구조 및 관련 문서 경로 확인

### 2단계: 현재 상태 파악 (`docs/memory/project-status.md`)

- 현재 개발 단계 및 완료/미완료 작업 확인
- 알려진 이슈 확인 (새 작업에 영향 여부 판단)

### 3단계: 최근 변경사항 파악

```bash
git log --oneline -10   # 최근 커밋 흐름 파악
git status              # 현재 작업 중인 변경사항 확인
```

### 4단계: 관련 소스 코드 확인

- 작업 대상과 직접 관련된 파일만 선택적으로 읽기
- `project-context.md`의 소스 구조를 참고하여 탐색 범위 결정

## 프로세스 흐름

```
컨텍스트 파악 → 명확한 질문 던지기 → 2~3가지 접근 방식 제안
     → 설계 섹션 제시 → [승인?] → 설계 문서 작성 → writing-plans 호출
                           ↑ 아니오, 수정
```

**최종 상태는 `writing-plans` 호출입니다.**
브레인스토밍 이후 호출할 수 있는 유일한 스킬은 `writing-plans`입니다.

## 상세 프로세스

**아이디어 이해하기:**

- 위 "컨텍스트 파악 순서"를 완료한 후 질문을 시작합니다.
- 아이디어를 구체화하기 위해 한 번에 하나씩 질문합니다.
- 가능하면 객관식 질문을 선호하되, 필요한 경우 주관식도 괜찮습니다.
- 목적, 제약 조건, 성공 기준을 이해하는 데 집중합니다.

**접근 방식 탐색하기:**

- 장단점이 있는 2~3가지 다른 접근 방식을 제안합니다.
- 권장 옵션을 먼저 제시하고 그 이유를 설명합니다.
- 현재 프로젝트의 기술 스택과 컨벤션에 맞는 방식을 우선 고려합니다.

**설계 제시하기:**

- 무엇을 만들지 이해했다고 판단되면 설계를 제시합니다.
- 각 섹션의 규모는 복잡도에 맞게 조정합니다.
- 아키텍처, 컴포넌트, 데이터 흐름, 에러 처리, 테스트 등을 다룹니다.
- 각 섹션을 마칠 때마다 지금까지의 내용이 맞는지 확인합니다.

## 설계 이후 단계

**문서화:**

- 검증된 설계를 `docs/plans/YYYY-MM-DD-<주제>-design.md`에 작성합니다.
- 설계 문서를 git에 커밋합니다.

**구현:**

- `writing-plans` 스킬을 호출하여 상세 구현 계획을 작성합니다.
- 다른 스킬을 호출하지 마십시오. `writing-plans`가 다음 단계입니다.

**상태 업데이트:**

- 작업 완료 후 `docs/memory/project-status.md`의 해당 항목을 업데이트합니다.

## 핵심 원칙

- **한 번에 질문 하나씩** — 여러 질문으로 사용자를 혼란스럽게 하지 않습니다.
- **객관식 선호** — 가능하면 주관식보다 답변하기 쉬운 객관식을 제공합니다.
- **철저한 YAGNI** — 모든 설계에서 불필요한 기능은 과감히 제거합니다.
- **대안 탐색** — 결정을 내리기 전에 항상 2~3가지 접근 방식을 제안합니다.
- **점진적 검증** — 설계를 제시하고 다음 단계로 넘어가기 전에 승인을 받습니다.
- **컨텍스트 우선** — 질문 전 반드시 컨텍스트 파악을 완료합니다.
