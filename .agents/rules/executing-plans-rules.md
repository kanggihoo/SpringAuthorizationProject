# 상세 계획 실행 및 구현 규칙 (Executing Plans Rules)

## 개요 (Overview)

이 규칙은 `writing-plans` 단계에서 확정된 상세 명세서(`docs/specs/stepN-*.md`)를 가이드 삼아, 실제 소스 코드를 작성하고 검증하며 기능을 완성해나가는 단계입니다.

핵심은 **"작은 단위의 실행과 확실한 검증"**입니다. 한꺼번에 모든 코드를 짜지 않고, 명세서에 정의된 각 컴포넌트나 로직을 단계별로 나누어 구현하며 매 단계마다 테스트 통과 여부를 확인합니다.

**시작 시 선언:** "상세 계획 실행(Executing Plans) 규칙을 사용하여 구현을 시작합니다."

## 🛠️ 구현 및 검증 프로세스

### 0단계: 환경 설정 및 필수 스킬 참조 (Environment Setup & Skill Indexing)

구현 단계의 안정성과 개발 표준을 준수하기 위해 코딩 시작 전 다음 두 가지 필수 스킬(Skill)을 문서 내 명시된 경로를 통해 반드시 인덱싱(조회 및 숙지)하고 그 지침에 따라 작업을 준비하십시오:

1.  **Git Worktree 설정 (`using-git-worktrees`)**
    - **참조 스킬:** `@[/Users/kkh/Desktop/javaMiniProject/.agents/skills/using-git-worktrees/SKILL.md]`
    - **목적:** 기존 작업 브랜치를 오염시키지 않고, 격리되고 안전한 환경에서 상세 계획 구현을 진행하기 위해 작업 트리를 우선적으로 구성합니다.

2.  **Spring Boot TDD 방법론 (`springboot-tdd`)**
    - **참조 스킬:** `@[/Users/kkh/Desktop/javaMiniProject/.agents/skills/springboot-tdd/SKILL.md]`
    - **목적:** 도메인, 리포지토리, 서비스, 컨트롤러 등 각 계층에 맞는 테스트 우선 접근법(Test-First)과 검증 방식을 정확하게 코드에 적용하기 위해 숙지합니다.

### 1단계: 명세서 로드 및 분석

1.  `docs/specs/` 폴더에서 해당 작업의 상세 명세서를 읽습니다.
2.  명세서의 내용을 비판적으로 검토하며, 구현 시 발생할 수 있는 잠재적 충돌이나 누락된 의존성이 없는지 확인합니다.
3.  의문점이 있다면 구현 시작 전에 반드시 사용자에게 질문하여 명확히 합니다.

### 2단계: 작업 단위 실행 (TDD Cycle)

명세서에 정의된 각 기능을 아래 순서에 따라 반복(Iteration)하며 구현합니다.

1.  **테스트 우선 작성(Test First):** 구현하려는 로직에 대한 실패하는 테스트 코드를 먼저 작성합니다.
2.  **테스트 실패 확인:** `./gradlew test` 터미널 명령어를 이용하여 해당 테스트가 기대대로 실패하는지 확인합니다.
3.  **최소 구현(Production Code):** 테스트를 통과시키기 위한 가장 단순하고 명확한 코드를 작성합니다.
4.  **테스트 통과 확인:** 다시 테스트를 실행하여 성공(PASS)하는지 확인합니다.
5.  **리팩토링 및 커밋:** `project-context.md`의 컨벤션을 준수하도록 코드를 정리하고, 의미 있는 단위로 `git commit`을 수행합니다.

### 3단계: 중간 보고 및 피드백

- 하나의 주요 컴포넌트나 로직 그룹(예: 엔티티와 리포지토리 완성 등)이 끝날 때마다 사용자에게 진행 상황을 보고합니다.
- **보고 내용:**
  - 무엇이 구현되었는가?
  - 어떤 테스트가 수행되었고 결과는 어떠한가?
- 사용자의 피드백을 반영하여 다음 단계로 넘어가거나 코드를 수정합니다.

## ⚠️ 중단 및 도움 요청 기준

다음 상황에서는 구현을 즉시 중단하고 사용자에게 도움을 요청합니다:

- **블로커 발생:** 예상치 못한 의존성 충돌, 라이브러리 버그, 또는 명세서의 지침이 실제 코드와 충돌할 때.
- **테스트 실패 지속:** 여러 번의 수정 후에도 테스트를 통과시키지 못할 때.
- **명세 불명확:** 특정 구현 디테일이 명세서에 누락되어 있어 임의의 판단이 필요할 때.

**절대로 추측하여 구현하지 마십시오. 모호할 때는 질문하는 것이 가장 빠릅니다.**

## 작업 완료 및 상태 업데이트

모든 명세 사항이 구현되고 테스트가 완료되면 다음을 수행합니다:

1.  **상태 문서 업데이트:** `docs/memory/project-status.md`를 열어 완료된 항목을 체크하고, 새로 발견된 이슈가 있다면 기록합니다.
2.  **최종 보고:** 전체 구현 결과와 테스트 요약을 사용자에게 보고합니다.
3.  **다음 단계 안내:** 마일스톤의 다음 작업이나 개선 사항을 제안합니다.
