# 구체적 구현 사양 및 계획 작성 규칙 (Writing Plans Rules)

## 개요 (Overview)

이 규칙은 이전 `brainstorming` 단계에서 도출된 추상적인 설계와 아이디어를, 실제 개발을 즉시 시작할 수 있는 수준의 **체계적이고 구체적인 명세서(Specification)**로 구체화하는 과정입니다.

테스트 코드 명령어 작성이나 커밋 메시지 작성과 같은 단순 행위 지침은 이 규칙에 포함되지 않으며, 오직 **"무엇을 어떻게 만들지에 대한 설계의 끝맺음"**에 집중합니다. 이 과정이 완료되면 실제 코드를 작성하는 `executing-plans` 과정으로 넘어갑니다.

**시작 시 선언:** "상세 구현 계획 작성(Writing Plans) 규칙을 시작하여 아이디어를 구체적인 명세서로 발전시키겠습니다."

---

## 💡 구체화를 위한 사용자와의 상호작용 규칙 (매우 중요)

AI는 명세서를 독단적으로 작성하지 않습니다. 막연히 세워진 아이디어를 구체화하기 위해 사용자와 협력하며, 다음 원칙을 따릅니다:

1.  **세부 사항 파악:**
    - 아이디어를 실제로 구현하기 위해 빈틈이 있는 곳(예: 데이터베이스 스키마, 연관 관계, 예외 처리 파이프라인, 구체적인 API 스펙 등)을 식별합니다.
2.  **2~3가지 선택지 제시:**
    - 단순히 "어떻게 할까요?"라고 묻기보다는, 기존 프로젝트 컨텍스트에 기반하여 가장 합리적인 옵션 2~3가지를 분석하고 제안합니다.
    - 각 선택지의 **장점과 단점, 그리고 AI가 권장하는 옵션**을 함께 제시하여 사용자의 결정 부담을 줄이고 빠른 진행을 돕습니다.
3.  **한 번에 하나씩 묻기:**
    - 사용자가 혼란스럽지 않도록, 한 번에 하나의 구체적인 주제(예: "데이터베이스 모델 구조", "에러 반환 포맷")에 대해서만 논의를 진행하고 확정 지은 후 다음 주제로 넘어갑니다.

## 작업 구조 (Workflow)

### 1단계: 베이스라인 탐색 및 로드

- `docs/memory/project-context.md` 및 시스템의 상태 문서를 먼저 읽어 아키텍처 및 코딩 컨벤션을 잊지 않도록 숙지합니다.
- `brainstorming` 단계에서 만들어진 초기 아이디어를 로드합니다.

### 2단계: 핵심 구성요소 딥 다이브 (Deep Dive Q&A)

- (위 "상호작용 규칙"에 따라) 사용자와 질의응답을 진행하며, 모호한 부분을 전부 확정 짓습니다.

### 3단계: 상세 명세서(Specification) 초안 제시

모든 정책과 스펙이 확정되면, 사용자가 확인하기 쉽게 명세서 초안을 제시합니다. 아래의 **범용적인 명세서 포맷**을 따르되, 기능의 성격에 따라 유연하게 가감합니다.

```markdown
# [주제/기능명] 상세 명세서

## 1. 목적 및 목표 (Goals)

- [이 기능이 달성하고자 하는 비즈니스 목표나 техни적 목표 한 줄 요약]

## 2. 핵심 정책 및 비즈니스 로직 (Core Policies)

- [이 기능이 동작하기 위한 주요 정책, 예외 조건, 제약 사항 등]

## 3. 데이터 및 모델 구조 (Data Models)

- [생성/변경되는 주요 테이블, 필드, 타입 및 연관 관계 (필요한 경우)]

## 4. 인터페이스 명세 (Interfaces)

- [사용자 UI, API Endpoints, Queue Message 등 해당 기능의 인풋/아웃풋 구조]
- [Request / Response 포맷 및 Validation 규칙]

## 5. 핵심 컴포넌트 설계 (Components Flow)

- [어떤 로직이 어떤 계층(Controller, Service, Repository 등)에 위치하는지에 대한 설명]
- [내부의 핵심 동작 순서 (1. ~ 2. ~ 3. ~)]

## 6. 예외 처리 정책 (Error Handling)

- [발생할 수 있는 주요 예외 상황과 기대하는 시스템/사용자 응답]
```

### 4단계: 사용자 최종 승인 (Validation)

- 작성된 초안을 검토하며 누락된 엣지 케이스(Edge case)가 없는지 최종적으로 묻고 반영합니다.

### 5단계: 사양 문서 저장

- 확정된 사양을 명세 문서로 저장합니다.
- **저장 경로:** `docs/specs/stepN-[주제].md` (현재 마일스톤 단계에 맞춰 step 번호를 부여하여 저장)
- 문서 저장을 완료합니다.

## 실행 핸드오프 (Execution Handoff)

명세서 문서가 저장된 후, AI는 사용자에게 다음과 같이 실제 작업 착수를 안내합니다:

**"상세 명세서가 `docs/specs/<파일명>.md`에 성공적으로 저장되었습니다. 이제 이 사양서의 뼈대와 규칙을 바탕으로 실제 테스트 코드, 프로덕션 코드를 작성할 완벽한 준비가 끝났습니다.**

**이제 `executing-plans` 규칙을 사용하여 실제 구현 프로세스를 시작하시겠습니까?"**
